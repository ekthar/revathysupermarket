# Smart config rendering and WhatsApp/SMS flags for admin panel

The admin feature flags panel gains two missing boolean flags (WhatsApp and SMS notifications), a new "Notifications & Messaging" category, and a `CONFIG_HINTS` metadata map that drives type-aware rendering for 15 config flags. The old `ConfigFlagCard` — which rendered every config value as a raw text/number input — is replaced by `SmartConfigFlagCard`, which consults `CONFIG_HINTS` to render dropdowns, number inputs with unit decorators (₹, km, %, etc.), time pickers, and checkboxes. Flags without hints fall back to the previous generic rendering.

**Watch for:** Number inputs coerce empty/invalid values to `0` via `Number(e.target.value) || 0`, which silently destroys legitimate zero values and makes it impossible to clear a field (confirmed). Config spread captures `config` at render time, so rapid edits to multiple fields on the same card race and can overwrite each other (confirmed). When hints exist for a flag, config keys not covered by hints become invisible in the UI (confirmed).

## High-level view

The change solves two distinct problems: visibility (WhatsApp/SMS flags weren't in the UI at all because they had no `FLAG_META` entry or category membership) and usability (config flags rendered as bare text inputs with no indication of valid values or units). The `CONFIG_HINTS` map is a pure-UI concern — it doesn't affect persistence or the API contract, only what widget the admin sees.

The rendering uses `defaultValue` on inputs and fires `updateFlag` on blur/onChange, persisting config immediately on every interaction. This creates a stale-closure problem when multiple fields are edited in quick succession, and means the UI won't reflect server-side normalization or rejection of values.

3. **Hints-only rendering hides unknown config keys** — When hints exist for a flag, only the keys listed in `CONFIG_HINTS` are rendered. If the database contains additional config keys not covered by hints (e.g., added by a migration), those keys become invisible and uneditable in the UI without any warning.
4. **No controlled state for hint inputs** — `defaultValue` makes inputs uncontrolled. If `updateFlag` triggers a re-render with new state, the displayed value doesn't update to reflect the persisted value, creating a visual desync until the component remounts.

## Zero-coercion in number inputs

The number input blur handler does:

```tsx
const parsed = Number(e.target.value) || 0;
updateFlag(flag.key, { config: { ...config, [cfgKey]: parsed } });
```

`Number("") || 0` and `Number("0") || 0` both produce `0`, so the user can never clear a field or intentionally set a value to zero and distinguish it from "empty." More critically, `Number("abc") || 0` silently writes `0` instead of rejecting the input. The generic fallback path has the same pattern. A safer approach: parse with `parseFloat`, check `isNaN`, and either reject invalid input or leave the value unchanged.

## Config spread and stale closures

`renderHintField` closes over `config` (captured from `flag.config` at render time). When a user changes two fields in quick succession — say, `senderReward` and `receiverReward` on the referral card — the second onChange/onBlur fires with `config` still reflecting the pre-first-edit state. The spread `{ ...config, [cfgKey]: newValue }` overwrites the first edit's key with its original value.

In practice this manifests when a user tabs through fields quickly or when `updateFlag` is slow to round-trip. Fix by maintaining a local `configDraft` state within `SmartConfigFlagCard` and debouncing the persistence call, or by using a functional updater that reads the latest state.

## Hints-only rendering hides unknown keys

When `CONFIG_HINTS[flag.key]` is defined, the rendering loop iterates `Object.entries(hints)` — not `Object.entries(config)`. Any config key present in the database but absent from the hints map becomes invisible. For example, if a future migration adds a `priority` field to `max_orders_per_hour`, it won't appear in the UI until someone adds it to `CONFIG_HINTS`. The generic fallback (which iterates config keys) only fires when no hints exist at all for that flag. A defensive approach would render hint-driven fields first, then append any remaining config keys not covered by hints using the generic input.

## Uncontrolled inputs and re-render desync

All hint-driven inputs use `defaultValue`, which React treats as the initial-mount value. If `updateFlag` succeeds and the parent re-renders `SmartConfigFlagCard` with new `flagState`, the input still displays whatever the user last typed — not the persisted value. If the API normalizes or rejects a value, the UI and state silently diverge. Converting to controlled inputs (`value` + local state) or adding a `key` prop tied to a version counter would fix this.
