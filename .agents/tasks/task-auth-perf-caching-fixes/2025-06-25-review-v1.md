# Auth throttling, TanStack Query layer, error boundaries and cache revalidation

This change addresses three distinct performance and reliability concerns: session validation is throttled to avoid per-request database hits, client-side data fetching is migrated to TanStack Query for caching and optimistic updates, and missing error boundaries and cache revalidation calls are patched in. The auth throttling moves the DB lookup from the session callback to the jwt callback with a 5-minute staleness window. The TanStack Query integration adds a QueryProvider at the app root and introduces mutation hooks for favorites, addresses, notifications, and admin orders. Error boundaries are placed at five routes. Cache revalidation is added to admin offer/promo-code mutation routes.

**Watch for:** The auth throttling creates a 5-minute window where revoked users retain access (confirmed). Staff users get empty permissions on initial sign-in until the first revalidation fires (confirmed). The `useToggleFavorite` hook lacks optimistic cache updates — it only invalidates on success, while FavoriteButton uses local state, creating divergent sources of truth (confirmed).

## High-level view

The auth performance fix moves the expensive DB revalidation into the JWT callback with a `lastValidated` timestamp, so subsequent requests within a 5-minute window just format token data in the session callback. A deactivated user or password-rotated user keeps their old session for up to 5 minutes. More critically, the initial sign-in path fetches `passwordVersion`/`authVersion` but not `permissions` or `isActive`, so staff users start with an empty permissions array and will be denied access to permission-gated features until the first revalidation cycle.

TanStack Query is integrated at both infrastructure and feature levels. Optimistic updates are implemented for notifications and addresses but not for favorites — the toggle mutation invalidates on success instead of updating the cache in place, which creates a dual-state problem with FavoriteButton's local `useState`. The `favorites-client.tsx` rollback-on-error logic restores the initial server-rendered list, discarding any other mutations that succeeded in the meantime.

The admin orders polling uses `refetchInterval: 6000` (6 seconds), which is aggressive for a multi-tab admin panel — 10 requests/minute per tab. The `updateOrderStatus` cache update fires unconditionally from the `onStatusChange` callback, potentially optimistically updating the cache even if the underlying POST failed inside `OrderStatusForm`.

## Auth throttling: initial sign-in permissions gap

The initial sign-in path sets `token.permissions = []` unconditionally. The DB query at sign-in only selects `passwordVersion` and `authVersion`:

```typescript
const versions = await prisma.user.findUnique({
  where: { id: user.id },
  select: { passwordVersion: true, authVersion: true }
}).catch(() => null);
```

Staff users who log in will have an empty permissions array until the first revalidation cycle runs (up to 5 minutes). Any permission-gated feature accessed in that window will fail authorization. The fix is to include `staffPermissions` and `isActive` in this query and populate them in the token immediately.

The revocation gap itself (deactivated users retaining access for 5 minutes) is an accepted trade-off for performance, but it's worth noting that the security check uses `Number(token.passwordVersion ?? 0)` for comparison. If the initial sign-in `.catch(() => null)` fires and `versions` is null, the token stores `passwordVersion: 0`. If the actual DB value is non-zero, the first revalidation will see a mismatch and invalidate the session, which is fail-safe behavior.

## Favorites: divergent optimistic state and stale rollback

The `useToggleFavorite` hook does not implement `onMutate` — it only invalidates the query cache on success:

```typescript
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: FAVORITES_QUERY_KEY });
},
```

`FavoriteButton` manages its own local `isFavorited` state and flips it on click. If the mutation throws, the catch block reverts local state. But the query cache never had the optimistic value, so when it eventually refetches, the UI could flash back and forth.

`favorites-client.tsx` has a more concrete bug. On error, it rolls back to `initialProducts`:

```typescript
} catch {
  setProducts(initialProducts);
  showToast("Failed to remove", "error");
}
```

If the user removed item A successfully, then removing item B fails, the rollback restores both A and B. The correct pattern is to snapshot `products` before the mutation and roll back to that snapshot (as the address mutations do with their `previous` variable).

## Admin orders: unconditional cache update from OrderStatusForm

The `updateOrderStatus` function directly writes to the query cache:

```typescript
function updateOrderStatus(orderId: string, status: keyof typeof statusLabels) {
  queryClient.setQueryData<AdminOrder[]>(ADMIN_ORDERS_QUERY_KEY, (current) =>
    current?.map((order) => (order.id === orderId ? { ...order, status } : order)) ?? []
  );
}
```

This is called from `OrderStatusForm`'s `onStatusChange` prop. If `OrderStatusForm` calls this callback before confirming the API response succeeded, the cache will reflect a status that the server may have rejected. The 6-second polling will eventually correct it, but the admin sees a momentary lie.

## Error boundaries exposing raw error messages

All five error boundaries display `error.message` directly:

```tsx
{error.message || "Failed to load account information. Please try again."}
```

Next.js 15 digests server-side errors in production (replacing messages with generic text), so server component errors won't leak. But client-side errors — thrown from fetch handlers, JSON parsing, or third-party libraries — will display their raw message to end users. Consider always showing the generic fallback and logging `error.message` only to the console (which the `useEffect` already does).
