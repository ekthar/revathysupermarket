# Customer Unavailable flow, iOS push fix, and heading in location tracking

Three delivery-experience concerns addressed in a single change: a two-step "customer unreachable" workflow with countdown timer and return-to-store capability, hardened iOS push notification options in the service worker, and plumbing the device compass heading through location updates to the customer-facing tracking view.

Watch for:
- **Timeout never persisted** (confirmed) — `POST /api/delivery/unavailable` sets `customerUnavailableAt` but never writes `customerUnavailableTimeout` to the database, relying entirely on the fallback `?? 300` in the return endpoint. If the timeout is ever made configurable per-order or per-store, the value will silently stay null.
- **Tracking route only shows location for OUT_FOR_DELIVERY** (confirmed) — the `/api/orders/[id]/tracking` route gates `deliveryPartnerLocation` on `order.status === "OUT_FOR_DELIVERY"`, so customers whose order is in `ARRIVING` or `CUSTOMER_UNAVAILABLE` status see no rider location at all. The SSE stream route does not have this gate.
- **No idempotency guard on the unavailable endpoint** (confirmed) — calling `POST /api/delivery/unavailable` twice on the same order would fail on the second call because status is no longer `ARRIVING`, but the 404 error message ("Active assigned order not found") is misleading rather than indicating the action was already taken.

## High-level view

The customer-unavailable flow adds two new `OrderStatus` enum values and two API routes implementing a linear state machine: ARRIVING → CUSTOMER_UNAVAILABLE → RETURNED_TO_STORE. The client enforces the countdown purely via local state, while the server independently enforces elapsed time before allowing the return. The timeout value is hardcoded at 300 seconds and never persisted to the database despite the schema providing a column for it.

The iOS push fix makes `requireInteraction: true` unconditional (previously payload-driven) and adds `renotify: true`. This fixes iOS delivery alerts but also makes every notification in the app persistent — admin panel alerts, customer status updates, everything — with no opt-out path.

Heading propagation threads `position.coords.heading` from the geolocation watcher through to both the SSE stream and the polling tracking endpoint. The `DeliveryLocationEvent` model already had a `heading Float?` column; the only missing link was the client POST body.

6. **`order.userId!` non-null assertion** — Both endpoints assert `order.userId!` without null-checking. If an order lacks a customer, the push call receives `undefined`, swallowed by `.catch(() => null)`. Add an explicit guard or skip the push when userId is absent.

## Customer Unavailable state machine and timeout enforcement

The `unavailable` endpoint's Prisma update only writes `status` and `customerUnavailableAt`:

```typescript
data: { status: "CUSTOMER_UNAVAILABLE", customerUnavailableAt: now }
```

The `customerUnavailableTimeout` column remains null. The return endpoint compensates with `order.customerUnavailableTimeout ?? 300`, so the system works today with a fixed 5-minute window. The schema explicitly provides the column for per-order configurability, but the data is never written.

The client-side countdown is driven by a `Map<orderId, timestamp>` stored in React state. If the page reloads mid-wait, the countdown is lost and the "Return to Store" button immediately enables (remaining = 0 because the order ID is no longer in the map). The server still blocks the early return with a 409, so this is a UX inconsistency rather than a correctness bug — the partner sees the enabled button, clicks it, and receives an error with `remainingSeconds`. Initializing the map from server state (order status + `customerUnavailableAt`) on mount would eliminate the confusion.

## Tracking route status gate excludes ARRIVING and CUSTOMER_UNAVAILABLE

The `/api/orders/[id]/tracking` endpoint has this condition:

```typescript
deliveryPartnerLocation: order.status === "OUT_FOR_DELIVERY" && ...
```

Once the order progresses to `ARRIVING` or `CUSTOMER_UNAVAILABLE`, the tracking response returns `deliveryPartnerLocation: null` even though the rider is still actively sharing location. The SSE stream route (`/api/orders/[id]/stream`) does not have this gate — it always returns location data when coordinates exist. Customers using the polling endpoint lose the map pin while those on SSE keep it.

## Service worker requireInteraction scope

The change replaces `payload.requireInteraction || false` with a hardcoded `true`. The tag is already per-order (`order-${payload.orderId}`) so `renotify: true` correctly causes iOS to re-alert on updated notifications for the same order. However, the unconditional `requireInteraction` applies to every notification type passing through this service worker — not just delivery partner alerts. Low-priority customer notifications ("Your order has been delivered") will now persist in the notification tray until swiped, which may not be the intended behavior. A middle ground: default to `true` but allow `payload.requireInteraction === false` to override.

## `as never` type casts in OrderEvent creation

```typescript
await prisma.orderEvent.create({ data: { orderId: order.id, status: "CUSTOMER_UNAVAILABLE" as never } });
```

After running `prisma generate` with the updated schema, the Prisma client will accept the new enum values natively. Shipping the `as never` casts means the compiler can't verify the string matches a valid enum member, defeating the typed ORM's purpose. These should be removed post-generation.
