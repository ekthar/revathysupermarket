# Feature flags, distance-aware ETA, order bill, and map redesign

This change adds 26 new feature flags (for a total of 30) with a categorized admin UI, replaces the static ETA estimation with a distance-aware algorithm gated behind an `eta_display_mode` feature flag (default: show ETA only after delivery partner assignment), introduces an order bill/invoice component on the tracking page, and redesigns the delivery map markers with colored, labeled icons and a solid emerald route line on positron tiles.

Watch for: the `ConfigFlagCard` generic editor allows admin users to type arbitrary values into config fields without validation, which could produce malformed config that downstream consumers don't expect (confirmed). The `<style>` element injected by the map into `document.head` leaks if the component remounts multiple times within a session (confirmed). The `distanceKm` field used for ETA defaults to 3 km when missing from the order record, which may produce inaccurate estimates for new orders lacking that field (likely).

## High-level view

The feature flag system moved from a hand-crafted per-flag admin UI to a data-driven categorized layout. Four flags with complex config (forced-accept, ring alerts, print alerts) retain custom cards; the remaining 26 render through a generic `ConfigFlagCard` that introspects the JSON config and generates editable inputs per key. The category grouping is static in code, so new flags added to the database won't appear until the category array is updated.

The ETA algorithm is now parameterized by distance, speed, and rush-hour multiplier. The client-side `calculateEta` respects the display mode: in "after_assignment" mode it only returns a value once the rider is en route; in "always" mode it estimates for all active statuses. The default 3 km fallback when `distanceKm` is absent shifts ETA values slightly from the previous hard-coded table.

The order bill component renders line items, totals, and a print button for any order that has items, regardless of delivery status.

Map markers are now visually differentiated: emerald green rider with an ETA badge, blue rounded-square customer marker with "Your Home" label, and a small grey store circle. Keyframe animations are injected into `document.head` once per mount and removed on cleanup, but this approach is fragile under strict mode.

1. **No config value validation in generic editor** — `ConfigFlagCard` accepts any user input and writes it directly to the flag's config JSON. Add type-aware validation or at minimum a schema per flag to prevent malformed config.

3. **Stale ETA when `distanceKm` is null** — Orders without `distanceKm` in the database get a 3 km default, which can significantly over- or under-estimate. Consider showing "Calculating..." instead of a potentially misleading number.

5. **SVG filter ID collisions** — `createRiderMarkerEl` and `createCustomerMarkerEl` use hard-coded filter IDs (`rider-shadow`, `dest-shadow`). If multiple map instances exist on the same page, these IDs will collide and reference the wrong filter.

## Generic config editor allows unchecked writes

The `ConfigFlagCard` component iterates over the keys of the current `config` JSON and renders an `<input>` per key. On blur it coerces the value using a simple typeof check against the existing value: numbers get `Number()`, booleans check for `"true"`, everything else stays as a string. No validation against a schema, no min/max bounds, no enum enforcement.

```tsx
let parsed: unknown = raw;
if (typeof cfgValue === "number") parsed = Number(raw) || 0;
else if (typeof cfgValue === "boolean") parsed = raw === "true";
updateFlag(flag.key, { config: { ...config, [cfgKey]: parsed } });
```

An admin could set `delivery_radius_km.value` to `-5` or `surge_pricing.multiplier` to `0`, and that would be persisted and served to clients. The server-side PUT handler (`/api/feature-flags`) doesn't validate config shapes either. This matters most for flags consumed at runtime (minimum order value, max orders per hour, delivery radius) where an invalid value could break business logic.

## ETA jump on first rider location fix

When the rider is en route with a known GPS position, ETA is computed from rider-to-customer distance at 18 km/h. When the rider is en route without a location (brief GPS gap), the system falls back to `estimateOrderEta(status, { distanceKm })` using the order-level stored distance. The gap between these two calculation paths could cause the displayed ETA to jump discontinuously when the rider's first location fix arrives — for example, from a stored-distance-derived 10 min to a real-position-derived 4 min.

The backward-compatible default (`distanceKm=3`) also shifts existing banner ETA values: `ORDER_RECEIVED` goes from the old hard-coded 20 min to 23 min (`8 + 10 + 5`). Any consumer calling `estimateOrderEta` without arguments sees this change.

## Style element leak in delivery map

The `<style>` element with `@keyframes rider-pulse` and `@keyframes map-pulse` is created in the initialization effect and removed in its cleanup return. In Next.js App Router with React strict mode in development, effects run twice. The second mount creates a second `<style>` tag before the first cleanup runs, leaving an orphaned tag in `document.head`. In production this likely only fires once, but it's a correctness gap that could surface as doubled animation intensity.

A related minor issue: the "load" event handler adds the route source/layer with empty coordinates, and the `deliveryPartnerLocation` effect also conditionally adds the source if not present. The latter path is dead code when the map always initializes first, but creates a timing-sensitive double-add risk if `deliveryPartnerLocation` is set before map "load" fires.

## Category array drift in admin UI

`FLAG_CATEGORIES` is a hard-coded array mapping category names to flag keys. If a new flag is seeded but not added to a category, it won't appear in the admin UI at all — it exists in the database but is invisible to the admin. There's no "uncategorized" catch-all rendering. Adding a final category that renders `flags.filter(f => !usedKeys.has(f.key))` would prevent silent drift.

## Order bill displays server-stored totals without cross-check

The `OrderBill` component receives `subtotal`, `deliveryFee`, `tipAmount`, and `total` as props. It does not re-derive subtotal from `sum(quantity * price)`. If the stored `subtotal` doesn't match the items (after a partial refund, substitution, or coupon applied server-side), the bill will show the item breakdown alongside a non-matching subtotal. Users may notice the arithmetic doesn't add up.
