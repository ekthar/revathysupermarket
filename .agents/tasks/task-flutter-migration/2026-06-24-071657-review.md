# Flutter Web frontend served alongside Next.js backend

A new Flutter Web app (`msm_app/`) is introduced as the customer-facing frontend, intended to replace the existing Next.js pages. The Flutter build output is placed in `public/app/` and served via a Next.js rewrite rule at `/app/*`. A dedicated `/api/flutter-auth` endpoint issues raw JWTs for the Flutter client's OTP-based login. The Flutter side uses Dio with a Bearer token interceptor, go_router with auth guards, Riverpod state management, and freezed model classes.

Watch for:
- **Auth token incompatibility (confirmed):** The flutter-auth endpoint issues its own raw JWT, but every existing API route authenticates via NextAuth's `auth()` which reads session cookies ŌĆö not Bearer headers. The Flutter app will authenticate successfully at `/api/flutter-auth` but get 401s on all subsequent API calls (`/api/orders`, `/api/account/*`, etc.).
- **CSRF middleware may block Flutter mutations (likely):** The middleware rejects all non-GET requests where origin doesn't match. Flutter Web on the same host should pass since the browser attaches `Origin`, but this hasn't been verified end-to-end. Native/Capacitor builds will hard-fail.
- **Manifest link mismatch (confirmed):** `web/index.html` references `/manifest.webmanifest` but the actual file is `manifest.json`.

## High-level view

The integration strategy is a "sidecar" approach: Flutter builds to static files dropped into Next.js's `public/app/` directory, with a fallback rewrite ensuring client-side routing works. This avoids running a separate server but creates a coupling between the Flutter build step and the Next.js deployment pipeline.

The auth bridge is the linchpin of the entire integration. It duplicates the phone-OTP logic from the existing NextAuth `phone-otp` credentials provider into a standalone endpoint that returns a raw JWT. No existing API route knows how to consume this JWT ŌĆö they all call `auth()` which returns a NextAuth session derived from cookies. Without a middleware or wrapper that decodes the Bearer token into a session-like object, the Flutter app is authenticated in name only.

The Flutter architecture follows standard patterns: Riverpod for DI and state, go_router for navigation, Dio for HTTP, freezed for immutable models. All screens are placeholder shells ŌĆö the app has plumbing but no UI implementation yet.

## Auth bridge and the session gap

The `flutter-auth` route at `app/api/flutter-auth/route.ts` replicates the phone-OTP flow from the existing NextAuth `phone-otp` credentials provider almost line-for-line ŌĆö `normalizeIndianPhone`, `verifyLatestOtp`, `prisma.user.upsert` with identical logic. The difference is the output: instead of producing a NextAuth session, it signs a raw JWT with `NEXTAUTH_SECRET` and returns it in the response body.

The Flutter client stores this token and attaches it as `Authorization: Bearer <token>` on every request via the Dio interceptor. But the existing API endpoints all authenticate by calling:

```typescript
const session = await auth();
if (!session?.user?.id) return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
```

`auth()` reads the NextAuth session from cookies. It never inspects the `Authorization` header. The Flutter app logs in successfully but every subsequent API call returns 401.

Two paths to fix this:

1. **Hybrid auth utility:** A function that tries `auth()` first, then falls back to verifying the Bearer token with `jsonwebtoken.verify(token, NEXTAUTH_SECRET)` and constructs a session-equivalent. Existing routes would call this instead of `auth()` directly.
2. **Issue a NextAuth cookie:** Have the flutter-auth endpoint set the NextAuth session cookie in the response. The Flutter web app, running same-origin, would carry the cookie automatically.

Option 2 is simpler for same-origin web. Option 1 is needed if the Flutter app also targets native mobile (where cookies aren't practical).

The 30-day expiry has no revocation mechanism. NextAuth sessions enforce `passwordVersion` and `authVersion` checks on every request ŌĆö if a user changes their password or an admin deactivates the account, the session becomes invalid. The raw JWT bypasses all of this. A deactivated user remains authenticated for up to 30 days.

## Middleware CSRF interaction

`middleware.ts` matcher includes `/api/:path*`. For every non-GET request, it calls `isSameOriginRequest`, which checks that the `Origin` header's host matches the request URL's host. Flutter Web runs within a standard browser frame, so same-origin requests should carry the correct `Origin`. However, if the app is ever used via Capacitor or deployed to a different domain, all mutations will return 403.

The `isSameOriginRequest` function returns `true` if no `Origin` header is present ŌĆö Dio on web goes through the browser's fetch API which does attach Origin for POST requests. This path is likely safe for web but needs explicit verification in a deployed environment.

## SITE_URL asymmetry between web and Android builds

The `build:flutter` script passes `--dart-define=SITE_URL=$NEXT_PUBLIC_SITE_URL`. This becomes the Dio `baseUrl` in `api_client.dart`. If `NEXT_PUBLIC_SITE_URL` is empty or unset, Dio makes relative requests (fine for same-origin web). The Android build hard-codes `--dart-define=SITE_URL=https://msmsupermarket.in`. This creates a divergence: web can work with any deployment URL via env vars, but Android is pinned to production. A staging or dev build of the APK would still hit production APIs.

## PWA manifest mismatch

The `web/index.html` links to `/manifest.webmanifest` but the source file is `msm_app/web/manifest.json`. After `flutter build web`, this will be output as `manifest.json`. The link won't resolve.

The manifest metadata is boilerplate: `"name": "msm_app"`, `"description": "A new Flutter project."`, `"background_color": "#0175C2"` (Flutter blue, not the brand orange `#FF6600`).

## Token storage on web

`FlutterSecureStorage` on web falls back to `window.localStorage` ŌĆö there's no web secure enclave. Combined with the CSP allowing `'unsafe-inline'` scripts, an XSS vector would yield direct token access. This is no worse than a non-HttpOnly cookie, but it means the "secure storage" abstraction provides no actual security benefit on this platform.

## Test coverage

A single test (`widget_test.dart`) verifies the app renders "Home Screen" text. Not tested: API client behavior on 401/network errors, provider state transitions, router guard redirects, auth flow end-to-end, or any widget in isolation.
