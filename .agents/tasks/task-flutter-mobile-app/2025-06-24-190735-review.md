# Flutter mobile migration for MSM Supermarket

This change adds a full Flutter mobile app alongside the existing Next.js web app, targeting both customers and delivery partners. It introduces a versioned mobile API layer (`/api/mobile/v1/`) with JWT bearer token authentication, retrofits existing delivery endpoints with hybrid auth (bearer-first, session-fallback), and scaffolds a complete Flutter project with Riverpod, GoRouter, Dio, and Firebase Cloud Messaging for push-driven delivery alerts. The approach preserves all server-side price calculations and order validation, keeping the mobile client thin on business logic.

**Watch for:** FCM device registration is missing `installationId` in the payload, which will cause 400s from the server's zod schema (confirmed). The auth interceptor retry creates a fresh Dio without interceptors, bypassing error handling (confirmed). Server-side FCM push dispatch is not implemented—assignment alerts only reach mobile via the 30-second reconciliation poll, not instant push (confirmed). Refresh token rotation has a scan-and-compare pattern that degrades under multiple tokens per device (confirmed).

## High-level view

The refresh rotation logic works correctly for single-device scenarios but has a linear bcrypt scan over all non-revoked tokens for a given deviceId. The logout endpoint's "single device" path actually revokes all tokens for the user, contradicting its API contract.

The delivery partner alert system has the client-side architecture for instant FCM-driven alarms (foreground alarm overlay, background high-priority notification, reconciliation polling), but the server never actually sends the FCM push. The admin assignment route only emits an SSE event (for the web UI) and creates the `AssignmentEvent` record. Until server-side Firebase Admin dispatch is added, mobile partners will experience up to 30 seconds of delay before seeing new assignments.

The auth interceptor has two flaws: the retry after refresh creates a raw Dio without configuration, and concurrent 401s during refresh are dropped rather than queued. The FCM device registration will fail every call because the payload is missing the required `installationId` field.

The existing delivery endpoints have been migrated from session-only auth to hybrid auth without changing business logic. Cart validation against server prices before checkout maintains the existing trust model, reusing the web checkout's `createAuthoritativeOrder` for identical pricing.

1. **No server-side FCM push dispatch** — The admin assignment route creates an `AssignmentEvent` and sends an SSE alert, but never sends an FCM push to the partner's registered device. The client expects `data.type == 'delivery_assignment'` messages via FCM. Add Firebase Admin SDK dispatch after creating the event.
2. **FCM device registration payload mismatch** — `FcmService._registerTokenWithBackend` sends `{token, platform}` but the server's `/devices` endpoint requires `{installationId, platform, token}`. Add `installationId` to the registration payload.
3. **Auth interceptor retry bypasses API client configuration** — The retry in `AuthInterceptor.onError` creates a raw `Dio()` without baseUrl, interceptors, or timeouts. Use the existing configured Dio instance or pass retry options through the handler instead.
4. **Concurrent 401s drop requests during refresh** — When `_isRefreshing` is true, additional 401 errors are passed through (rejected) instead of queued for retry after refresh completes. Implement a request queue that waits for the refresh to complete.
5. **MobileRefreshToken tokenHash index is unused** — The refresh endpoint finds candidates by `(deviceId, revokedAt=null, expiresAt>now)` and then iterates bcrypt comparisons. The `tokenHash` index is never used in a WHERE clause. Remove it or switch to a non-bcrypt hash for lookup.
6. **Refresh token linear scan degrades** — If a device accumulates many non-revoked tokens (from bugs or missed revocations), the sequential bcrypt compare loop becomes expensive. Add a hard limit or clean up old tokens on each rotation.
7. **Logout single-device revokes all tokens** — The "refreshToken" branch in `/auth/logout` revokes all tokens for the user instead of just the one matching the provided refreshToken. This logs out all devices on any single-device logout.
8. **Missing alarm audio asset** — `assets/audio/delivery_alarm.mp3` has only a `.gitkeep` placeholder. The alarm screen will appear but be silent in production until the actual audio file is added.

## JWT auth and token rotation

The refresh flow in `app/api/mobile/v1/auth/refresh/route.ts` loads all non-revoked, non-expired tokens for a given `deviceId`, then sequentially bcrypt-compares the incoming token against each candidate:

```typescript
const candidates = await prisma.mobileRefreshToken.findMany({
  where: { deviceId, revokedAt: null, expiresAt: { gt: new Date() } },
});
for (const candidate of candidates) {
  const matches = await compareRefreshToken(refreshToken, candidate.tokenHash);
  if (matches) { matchedToken = candidate; break; }
}
```

This is O(n) bcrypt operations where n is the number of active tokens per device. If token cleanup fails or a client retries refresh without consuming the new token, stale tokens accumulate with no TTL-based hard deletion. The `tokenHash` index in the migration is misleading—bcrypt hashes aren't used for direct lookup, so the index provides no query benefit.

The logout endpoint's single-device path has a comment acknowledging it revokes all tokens ("since we can't easily find by hash match in bulk"), but the effect is that logging out one device logs out every device for that user. This contradicts the API contract implied by accepting a `refreshToken` body field (which suggests targeted revocation).

## Auth interceptor retry flaw

When the auth interceptor receives a 401 and successfully refreshes the token, it retries the original request:

```dart
final dio = Dio();
final response = await dio.fetch(options);
return handler.resolve(response);
```

This new Dio instance has no configured baseUrl, timeouts, or interceptors. The request URL is already resolved (fine), but the retry won't have error handling, and if it gets a 401 due to a race condition it throws an unhandled DioException.

When `_isRefreshing` is already true and another 401 arrives, the interceptor passes the error through. Concurrent requests that hit 401 simultaneously will only retry one; the rest fail to the caller. The standard solution is a Completer-based queue where pending requests wait for the refresh to finish.

## FCM registration payload gap and missing server dispatch

The server's `/api/mobile/v1/devices` route validates with:
```typescript
z.object({
  installationId: z.string().min(1),
  platform: z.enum(["android", "ios"]),
  token: z.string().min(1),
})
```

`FcmService._registerTokenWithBackend` sends only `{token, platform}` without `installationId`. This fails zod validation with a 400 on every token registration.

More fundamentally, no server-side code actually sends FCM pushes. The admin assignment route (`app/api/admin/orders/[id]/delivery/route.ts`) calls `sendDeliveryAlert()`, which is SSE-only (writes to in-memory `ReadableStreamDefaultController` sets for connected web clients). The `AssignmentEvent` record is created with `fcmSent: false` and nothing ever sets it to true. The entire FCM client infrastructure (background handler, foreground handler, `onAssignmentReceived` callback) is ready but will never fire until a Firebase Admin SDK integration is added server-side to actually dispatch the message to registered device tokens.

The reconciliation poll (every 30 seconds) serves as a working fallback, but the stated design intent of "instant alarm on push" requires the server-side dispatch.

## Assignment event deduplication

The client-side `AssignmentRepository` maintains `_alarmTriggeredEventIds` in memory only. On app restart, this set is cleared, causing reconciliation to re-alarm for any still-unacknowledged assignment. This is correct behavior—a delivery partner who killed the app mid-alarm should be re-alerted. The server-side acknowledgment (`acknowledgedAt` timestamp) is the source of truth for whether an alarm should fire.

## Test coverage gaps

Not tested: hybrid auth adapter integration (no test hits delivery endpoints with a Bearer token), FCM device registration (would immediately fail due to payload mismatch), support ticket endpoints, account profile endpoints, GoRouter redirect logic, and server-side FCM dispatch (which doesn't exist yet).
