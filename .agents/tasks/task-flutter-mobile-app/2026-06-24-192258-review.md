# Flutter mobile migration for MSM Supermarket (v2)

This change adds a full Flutter mobile app alongside the existing Next.js web app, targeting both customers and delivery partners. It introduces a versioned mobile API layer (`/api/mobile/v1/`) with JWT bearer token authentication, retrofits existing delivery endpoints with hybrid auth (bearer-first, session-fallback), scaffolds a complete Flutter project with Riverpod, GoRouter, Dio, and Firebase Cloud Messaging for push-driven delivery alerts, and now includes server-side FCM dispatch via Firebase Admin SDK. The approach preserves all server-side price calculations and order validation, keeping the mobile client thin on business logic.

**Watch for:** Profile update allows changing phone/email without re-verification, enabling account takeover via phone number swap (confirmed). The `sendFcmToUser` OAuth token is generated fresh on every push with no caching, adding ~500ms latency per dispatch and rate-limit risk with Google's token endpoint (confirmed). The `AuthRepository` refresh flow uses a bare `Dio()` instance that bypasses the configured `baseUrl` prefix, meaning refresh calls construct the full URL themselves but without shared timeouts or certificate pinning (likely). The non-blocking token cleanup in the refresh endpoint is fire-and-forget with no `await`, so if the response stream closes before the delete executes, stale tokens survive indefinitely (confirmed).

## High-level view

All eight v1 issues have been addressed. FCM dispatch now happens server-side via the Firebase Admin SDK HTTP v1 API, device registration includes `installationId`, the auth interceptor uses the configured Dio and queues concurrent 401s with a Completer, the refresh token scan is capped at 5 candidates with cleanup on rotation, logout performs targeted single-token revocation, and the misleading `tokenHash` index was replaced with a compound index on `(deviceId, revokedAt, expiresAt)`.

The FCM dispatch implementation (`lib/fcm-admin.ts`) creates a fresh Google OAuth token on every `sendFcmToUser` call. Under load (multiple assignments in a burst), this hammers Google's token endpoint unnecessarily and adds latency to every push.

The profile update endpoint (`/account PATCH`) accepts phone and email changes through a simple zod schema without requiring OTP re-verification or confirming ownership of the new phone/email. Since phone numbers are the primary login credential, this is effectively an account-identity mutation without proof of ownership.

<details>
<summary>Issues (4)</summary>

1. **Profile phone/email change without re-verification** — The `/account` PATCH endpoint accepts `phone` and `email` fields and writes them directly to the user record. Since phone is the login credential, a compromised session can permanently reassign the account. Require OTP verification for phone changes and email confirmation for email changes.
2. **FCM OAuth token not cached** — `getFirebaseAccessToken` generates a fresh JWT and exchanges it for an access token on every push. Google access tokens are valid for 3600 seconds. Cache the token in-memory (module-level variable with expiry check) to avoid redundant token exchanges and potential rate limiting.
3. **Non-blocking refresh token cleanup may never execute** — The `deleteMany` call in the refresh endpoint is deliberately not awaited (fire-and-forget). If the Next.js edge/serverless runtime tears down the handler after sending the response, the promise may be garbage-collected before execution. Use `waitUntil` (Next.js 14+) or await the cleanup.
4. **AuthRepository uses unconfigured Dio for refresh** — The `AuthRepository` creates its own `Dio()` instance (or accepts an injected one) separate from the `ApiClient`'s configured Dio. This means refresh requests don't inherit certificate pinning, proxy settings, or consistent timeouts configured on the main client. Pass the same Dio instance or at minimum mirror the `BaseOptions`.

## FCM server dispatch and OAuth token lifecycle

`getFirebaseAccessToken` runs the full OAuth2 JWT-bearer exchange on every invocation of `sendFcmToUser`. The returned access token is valid for one hour (`exp: now + 3600`), but nothing stores it between calls. In a serverless environment each cold-start creates a new token anyway, but within a warm instance handling multiple assignments (e.g., admin assigning 5 orders in quick succession), each push incurs the full RSA sign + HTTP round-trip to Google. A module-level cache with an expiry check would eliminate the redundant calls:

```typescript
let cachedToken: { token: string; expiresAt: number } | null = null;

async function getFirebaseAccessToken(...) {
  if (cachedToken && Date.now() / 1000 < cachedToken.expiresAt - 60) {
    return cachedToken.token;
  }
  // ... existing logic ...
  cachedToken = { token: accessToken, expiresAt: now + 3600 };
  return accessToken;
}
```

## Profile update identity mutation

The account PATCH endpoint:

```typescript
const updateProfileSchema = z.object({
  name: z.string().min(2).optional(),
  email: z.string().email().optional(),
  phone: z.string().min(8).optional(),
}).strict();
```

This schema feeds directly into `prisma.user.update`. The `phone` field is the primary login identifier (used in `normalizeIndianPhone` → user lookup). Changing it without re-verifying means: (a) a stolen access token (15-min window) can permanently hijack the account by changing the phone number, (b) the old phone owner loses access with no notification. The email field has the same issue if email-based login is later enabled.

## Refresh token cleanup race

In `app/api/mobile/v1/auth/refresh/route.ts`:

```typescript
// Clean up old revoked/expired tokens for this device to prevent accumulation
prisma.mobileRefreshToken.deleteMany({
  where: { ... },
}).catch(() => null); // Non-blocking cleanup
```

The promise is not awaited and not passed to any lifecycle hook. In Vercel's serverless model, the runtime can freeze the execution context after the response is sent, meaning this delete may or may not execute depending on timing. The v1 concern about token accumulation was the bcrypt scan; the fix (take:5) bounds the scan cost, but the cleanup needs to actually execute to prevent unbounded table growth over time. Using Next.js `waitUntil` or simply awaiting it (the latency is negligible for a DELETE with an index hit) would make this reliable.

## AuthRepository Dio configuration asymmetry

`AuthRepository` accepts a `Dio? dio` parameter and defaults to `Dio()` (no base options):

```dart
AuthRepository({
  required TokenStorage tokenStorage,
  required EnvironmentConfig config,
  Dio? dio,
}) : ...
     _dio = dio ?? Dio();
```

Refresh and logout calls construct URLs manually (`'${_config.baseUrl}/auth/refresh'`), which works for path resolution, but this Dio instance has no connect/receive timeouts, no certificate pinning, and no interceptors. If the main `ApiClient` is later configured with pinning or proxy, auth-critical requests (refresh, logout) bypass that configuration. The `AuthInterceptor` correctly receives the configured Dio via constructor injection, but the `AuthRepository` that performs the actual refresh HTTP call uses its own unconfigured instance.

## Test coverage gaps

Backend tests validate pure business logic without database calls (appropriate for CI speed but no integration coverage of Prisma queries or hybrid auth headers). Flutter tests cover repositories and widgets in isolation.

Not tested: profile update phone-swap scenario, FCM dispatch end-to-end (OAuth + HTTP), refresh token cleanup execution guarantee, GoRouter redirect logic with actual navigation stack, delivery endpoint hybrid auth with a real Bearer token.
