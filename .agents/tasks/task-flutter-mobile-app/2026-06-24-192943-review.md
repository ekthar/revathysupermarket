# Flutter mobile migration for MSM Supermarket (v3)

This change adds a complete Flutter mobile app alongside the existing Next.js web application, serving both customers and delivery partners. It introduces a versioned mobile API layer (`/api/mobile/v1/`) with JWT bearer token authentication, server-side FCM dispatch via Firebase Admin SDK, hybrid auth on existing delivery endpoints (bearer-first, session-fallback), and a full Flutter project with Riverpod, GoRouter, Dio, Freezed models, and Firebase Cloud Messaging. All server-side price calculations and order validation remain authoritative — the mobile client is intentionally thin on business logic.

**Watch for:** The `AuthRepository.refreshToken()` constructs the URL by prepending `_config.baseUrl` to `/auth/refresh`, but `_dio` already has `baseUrl` configured in its `BaseOptions` — Dio's path-override behavior masks this today, but it leaves dead configuration and will break if the Dio instance is ever shared with the `ApiClient` (confirmed). The `sendFcmToUser` fire-and-forget `.then()` in the admin assignment route may not complete on serverless teardown, making the `fcmSent` flag unreliable as an observability signal (likely).

**Verdict**: APPROVED

## High-level view

All eight v1 issues and all four v2 issues have been addressed. Profile PATCH is locked to `name` only. FCM OAuth token is cached module-level with a 60-second safety margin. Refresh token cleanup is awaited. The auth interceptor uses the injected Dio and queues concurrent 401s via a Completer. The refresh scan is capped at 5 candidates. Logout performs targeted single-token revocation.

The FCM integration is end-to-end: admin assignment creates an `AssignmentEvent`, dispatches an FCM data message, marks `fcmSent`/`fcmSentAt` on success, and the Flutter client handles foreground and background delivery with full-screen alarm overlay, queued events, and reconciliation polling as fallback.

The two remaining concerns are non-blocking: the `AuthRepository` URL construction is a latent design inconsistency (works today due to Dio's full-URL override behavior), and the FCM dispatch tracking is best-effort by design with the reconciliation poll guaranteeing delivery partner notification regardless.

<summary>Issues (2)</summary>

1. **AuthRepository double-prefixes URL on refresh and logout** (confirmed) — `_dio` is constructed with `baseUrl: config.baseUrl` (e.g., `http://10.0.2.2:3000/api/mobile/v1`), but `refreshToken()` and `logout()` call `_dio.post('${_config.baseUrl}/auth/refresh', ...)`. Dio treats paths starting with `http` as absolute, overriding `BaseOptions.baseUrl`, so this works today. But it means `BaseOptions.baseUrl` is dead config, and the `AuthInterceptor.skipAuthPaths` check (substring match on `options.path`) matches against a full URL rather than a relative path. If this Dio instance is ever shared with `ApiClient`, the interceptor path logic will break. Fix: use relative paths (`/auth/refresh`, `/auth/logout`, `/auth/login`) consistently.
2. **FCM dispatch tracking is fire-and-forget without lifecycle guarantee** (likely) — `sendFcmToUser(...).then(...)` in the admin assignment route is not awaited. On Vercel serverless, the runtime can freeze after sending the response, so the `.then()` callback updating `fcmSent`/`fcmSentAt` may never execute. The push message itself will typically complete before teardown, but the database flag becomes unreliable. If this flag is used for ops dashboards or retry logic, await the dispatch or use `waitUntil`.

## AuthRepository URL construction

The `AuthRepository` creates its Dio with a configured baseUrl:

```dart
_dio = dio ?? Dio(BaseOptions(
  baseUrl: config.baseUrl, // 'http://10.0.2.2:3000/api/mobile/v1'
  ...
));
```

But all method calls use the full URL:

```dart
await _dio.post<Map<String, dynamic>>(
  '${_config.baseUrl}/auth/refresh', // Full URL — Dio uses it as-is
  data: { ... },
);
```

Dio's behavior: if the path starts with `http://` or `https://`, it ignores `BaseOptions.baseUrl` and uses the path as the full URL. So the request goes to the correct endpoint. The issue is architectural: `BaseOptions.baseUrl` is dead configuration, and if this Dio instance is later wired through the `AuthInterceptor` (which uses `options.path.contains('/auth/login')` to skip auth headers), the substring match will still work against the full URL — but the intent becomes fragile. The `_login()` method uses the same full-URL pattern, so all three auth calls are consistent within `AuthRepository`. Since `AuthRepository` uses its own private Dio (not the `ApiClient`'s intercepted Dio), there is no runtime failure today.

## FCM dispatch lifecycle in admin assignment

```typescript
sendFcmToUser(parsed.data.deliveryPartnerId, {
  type: "delivery_assignment",
  eventId,
  orderId: id,
  orderNumber: order.orderNumber,
}).then(async (sent) => {
  if (sent) {
    await prisma.assignmentEvent.update({
      where: { eventId },
      data: { fcmSent: true, fcmSentAt: new Date() },
    }).catch(() => null);
  }
}).catch(() => null);
```

The push dispatch is deliberately non-blocking (don't slow the admin response). The FCM HTTP request will usually complete before the response is sent since the admin route awaits the `AssignmentEvent.create` and audit log before returning. The `.then()` callback that marks `fcmSent: true` is the part at risk — it runs after the response may already be streaming. The reconciliation poll (every 30 seconds) guarantees the delivery partner receives the assignment regardless of whether the push lands, so this is an observability gap rather than a user-facing failure mode.
