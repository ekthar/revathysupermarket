# Premium UI redesign for Flutter mobile app

A comprehensive visual and architectural overhaul of the Flutter mobile app touching 39 files (+13,811 / -2,763 lines). The change introduces Google Fonts (Inter Tight + Manrope), shared animation widgets (shimmer, animated fade-in, gradient card), a persistent bottom navigation shell via GoRouter's StatefulShellRoute, Riverpod providers for data fetching across all feature domains, and a consistent emerald-gradient design language. Every customer-facing screen and all delivery partner screens are redesigned with premium animations, loading states, and error handling.

**Watch for:** Unchecked cast of `state.extra as Product` in the router will throw at runtime if the route is deep-linked without extras (confirmed). Account/products providers silently swallow all errors and return empty defaults, making production debugging difficult (confirmed). The `_shellNavigatorKey` is declared but never used, while the cart badge in the bottom nav is hard-coded to 0 rather than connected to `cartItemCountProvider` (confirmed). `AnimatedBuilder` is used in two files but does not exist in Flutter's widget library — the correct widget is `AnimatedBuilder` only in Flutter 3.22+; for earlier versions this will fail to compile (likely).

## High-level view

The router migrates from flat GoRoute list to a StatefulShellRoute with 5 customer tabs, routes outside the shell for detail/checkout screens pushed over the nav bar, and custom page transitions using the `animations` package. One unsafe cast and a dead navigator key remain.

Data fetching shifts from prop-drilling to Riverpod FutureProviders (autoDispose) for products, banners, categories, cart, orders, wallet, loyalty, favorites, addresses, notifications, and support tickets. The account providers catch all exceptions and return defaults, which means API failures are invisible to users and un-debuggable in production. The home screen providers properly propagate errors to the UI, creating an inconsistency.

Shared UI widgets (ShimmerWidget, AnimatedFadeIn, GradientCard, BottomNavShell) form a reusable kit consumed across all screens. The AnimatedFadeIn widget uses `AnimatedBuilder` which may not compile on older Flutter SDK versions.

The checkout screen contains hard-coded mock address data and a client-side-only promo code validation (accepts only "SAVE10" and "FRESH20"). These are placeholders that will not function with real data.

## Router architecture and StatefulShellRoute

The problematic pattern is at line 237:

```dart
final product = state.extra as Product;
```

If a user navigates to `/customer/home/product` via a deep link, `state.extra` is null, producing a `TypeError` at runtime. This is a crash path, not just a degraded experience. The fix is to check for null and either redirect or show an error state. No other route in the file uses this unsafe extras pattern — they all extract path parameters.

The `_shellNavigatorKey` on line 71 is declared as a `GlobalKey<NavigatorState>()` but never passed to the `StatefulShellRoute` constructor. It should be removed to avoid confusion about which navigator key governs shell navigation.

## Riverpod provider design and error handling

The critical gap is error visibility. Every account provider and the `productByIdProvider` catch all exceptions and return empty/null values:

```dart
} catch (_) {
  return const WalletData();
}
```

A misconfigured API URL, expired token, or network timeout all produce the same result: the user sees an empty list or zero balance with no indication that something went wrong. The home screen providers (`featuredProductsProvider`, `bannersProvider`, `categoriesProvider`) do *not* catch errors, allowing Riverpod's `AsyncValue.error` state to propagate and display the `_ErrorRetrySection` widget. This inconsistency means the home screen has error UX but wallet/loyalty/favorites/addresses/notifications/support screens do not — they just show empty states that look like the user has no data.

The `autoDispose` modifier on all providers means any time a screen is unmounted (tab switch in shell route), the provider is disposed and must refetch on return. For the home screen this causes a shimmer flash every time the user navigates away and back. The `cartProvider` avoids this correctly by not using autoDispose — the same pattern should be applied to home/categories data.

## Animation widget compatibility

Both `AnimatedFadeIn` (line 95) and `ProductCard` (line 165) use `AnimatedBuilder` as their rebuild widget. `AnimatedBuilder` has existed in Flutter since the beginning under that exact name in `package:flutter/widgets.dart`, so this should be fine — but the concern is worth verifying against the project's declared minimum Flutter SDK constraint, since the `animations` package (v2.0.11) requires Flutter 3.16+.

The `AnimatedFadeIn` widget uses `Future.delayed` for stagger timing, which means animations won't respect `timeDilation` during widget tests. Tests that rely on `pumpAndSettle` may complete before staggered animations begin.

## Theme unused semantic colors

The `_warning`, `_info`, and `_success` colors are defined in AppTheme but never referenced in the theme data or exposed as getters. Screens that need these colors (like order status badges) hard-code their own color values rather than referencing the theme constants, which means a palette change requires hunting through individual screens.

## Checkout step indicator and data connectivity

The checkout screen hard-codes its address list:

```dart
final List<Map<String, dynamic>> _addresses = [
  {'id': '1', 'label': 'Home', ...},
  {'id': '2', 'label': 'Work', ...},
];
```

This should consume the `addressesProvider` from the account providers. The promo validation is also client-side only (`code.toUpperCase() == 'SAVE10'`), which will need a backend endpoint. The checkout does correctly read cart state from `ref.watch(cartProvider)` for the order summary, so the cross-feature wiring pattern exists and just needs extending.

## Delivery partner screens — architectural inconsistency

The delivery partner screens (dashboard, order detail, pickup, damage report, collection, completion) remain `StatefulWidget` + manual `setState` rather than migrating to Riverpod. They continue to receive `apiClient` as a constructor parameter. This creates two data-fetching paradigms in the same app that will diverge further over time. If Riverpod is the chosen architecture, these screens should migrate eventually.

## Test coverage gap

Only two test files are updated. No tests exist for:

- Cart provider behavior (add, remove, clear, validate)
- Order/account providers
- Bottom navigation shell
- Product detail screen
- Checkout flow
- Any delivery partner screen
- Animation widgets (AnimatedFadeIn, GradientCard, ShimmerWidget)
- Router deep-link handling (the unsafe cast path)

The home_screen_test correctly demonstrates ProviderScope overrides for async providers — a pattern that should be replicated across the untested features.
