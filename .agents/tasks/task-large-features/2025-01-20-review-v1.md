# Packing staff alert, product cost/brand fields, and mobile sidebar groups

Three features land in a single batch: a full-screen packing assignment push alert in the React Native staff app (FCM handler, notifee channel, alarm screen with 30s countdown), cost price and brand fields wired through the admin product form and API, and a mobile-friendly grouped sidebar replacing the flat 2-column grid. The work follows existing patterns closely — the packing alert is modeled on the delivery alert, the product fields replicate the pattern of other optional fields, and the sidebar reuses the existing `NavItem.group` field.

**Watch for:** No notifee event handler routes the `fullScreenAction` to the packing alert screen — the notification fires but nothing navigates the user to `alert/packing/[eventId]` (confirmed). The `handleReject` inside the countdown `useEffect` closure captures a stale reference and can fire the rejection dialog after the component has already acted (confirmed). The background message handler duplicates the foreground switch-case logic without sharing code (confirmed, low severity).

## High-level view

The packing alert mirrors the delivery alert architecture: FCM receives a `packing_assignment` type, notifee displays a full-screen notification on a dedicated high-importance channel, and a new screen at `alert/packing/[eventId]` plays an alarm, counts down 30 seconds, then auto-rejects. The critical missing piece is the navigation bridge — nothing listens for notifee foreground/background events to call `router.push("/alert/packing/...")` when the notification fires or is tapped. The delivery alert has the same gap in this codebase, which means either there's an untracked native module handling it or both alerts rely solely on `fullScreenAction` launching the activity (which does not route to a specific Expo Router screen without a linking configuration).

The product form additions are straightforward — Zod schema, POST/GET APIs, add form, and edit modal all carry the new fields through consistently. The PATCH handler needs no changes because it already spreads validated fields into the Prisma update.

The sidebar refactor replaces a flat two-column link grid with nested `<details>` elements grouped by `NavItem.group`. The active group auto-opens. Purely visual, no API surface.

## Missing navigation bridge for packing notification tap

The `showPackingAlert` method in `notifee.ts` sets `fullScreenAction.id = "accept_packing"` and `pressAction.id = "view_packing"`, and declares inline notification actions for Accept/Reject. However, no code in the app listens for notifee events to translate these action IDs into navigation calls. Neither `onForegroundEvent` nor `onBackgroundEvent` from `@notifee/react-native` is registered anywhere in the codebase.

On Android, `fullScreenAction` with `launchActivity: "default"` opens the app's main activity, which starts Expo Router at the root. Without a linking configuration that maps notification data to a route, the user lands on whatever screen the auth-redirect logic sends them to — not the alert screen. The screen exists at `alert/packing/[eventId]` and is registered in the Stack, but there's no trigger to navigate there.

This means the packing alert screen will never be reached in practice unless there's an untracked native intent filter (not present in the diff or existing code).

## Stale closure and race condition in countdown

```tsx
useEffect(() => {
  alarmService.startAlarm();
  const interval = setInterval(() => {
    setCountdown((prev) => {
      if (prev <= 1) {
        clearInterval(interval);
        handleReject(); // <-- stale reference
        return 0;
      }
      return prev - 1;
    });
  }, 1000);
  return () => { clearInterval(interval); alarmService.stopAlarm(); };
}, []);
```

`handleReject` is captured at mount time. If the component re-renders (e.g., `processing` state changes from Accept being tapped), the interval still holds the original closure. The timer fires `handleReject()` which calls `setProcessing(true)` and shows an Alert dialog even though `handleAccept` already navigated away. The cleanup function runs on unmount, but if `handleReject` fires in the same tick as the countdown reaching zero, it races with any in-progress Accept.

The delivery alert screen (`alert/[eventId].tsx`) has the same pattern — this is a pre-existing issue replicated into new code. Flagging it here because the new screen is an opportunity to fix the pattern: store `handleReject` in a ref, or guard both handlers with an `isHandled` ref that prevents double-action.

## Background handler duplication

The new `setBackgroundMessageHandler` at module scope in `fcm.ts` duplicates the entire switch-case from `handleForegroundMessage`. Both paths route `delivery_assignment`, `packing_assignment`, `order_rejected`, and the default case identically. If a new message type is added to one and not the other, they'll silently diverge. A single `routeIncomingMessage(message)` function called from both locations would eliminate this drift risk.
