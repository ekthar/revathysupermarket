# Live order tracking, dark mode repair, and mobile animation polish

This change adds a full-screen live order tracking page (`/track/[id]`) with SSE real-time updates, fixes dark mode inconsistencies across the customer-facing app, and layers in Framer Motion micro-interactions (bottom-nav `layoutId` transitions, scroll-linked header shrink, staggered card animations, a haptic-feedback button primitive). The tracking page is the largest single addition: a premium mobile-first screen with rider info, map placeholder, ETA countdown, delivery OTP display, and a 4-step timeline stepper.

**Watch for:** SSE connection has no reconnection logic — a single `onerror` permanently closes the stream (confirmed). The `LiveOrderBanner` polls `/api/orders` every 30 seconds and scans all orders client-side to find active ones, which will scale poorly once a user accumulates order history (confirmed). The delivery OTP is surfaced in the client HTML for statuses that may be premature (possible — depends on when `OUT_FOR_DELIVERY` is set relative to rider proximity).

## High-level view

The tracking page follows a server-component-fetches, client-component-renders split. The server side enforces ownership (user must own the order or be staff) and projects only the fields needed for the UI; the client component then opens an SSE connection for live status and rider location updates. ETA is derived from Haversine distance with an assumed 18 km/h speed, falling back to static per-step estimates when no rider location is available.

Dark mode fixes are mechanical but broad: seven files replace the hardcoded `bg-[#F7F7FA]` with the CSS-variable-backed `bg-background`, and explicit `dark:` variants are added to cards, text, borders, and interactive states that previously only had light palette classes. A dark body gradient and dark variants for the custom `.ios-squircle-card` and `.ios-map-placeholder` classes round out the global layer.

The mobile smoothness additions are opt-in primitives (`AnimatedPageWrapper`, `StaggeredContainer`, `HapticButton`) plus targeted integration: the bottom nav gains a `layoutId`-driven active-tab pill that morphs between tabs with a spring, the header height shrinks on scroll via `requestAnimationFrame`-throttled scroll progress, and the recent-orders carousel uses staggered spring entry.

<summary>Issues (5)</summary>

1. **SSE stream never reconnects** — `EventSource.onerror` calls `source.close()` with no retry. Add exponential backoff reconnection or use the built-in retry semantics of SSE (server sends `retry:` field).
2. **Banner polls all orders to find active one** — `LiveOrderBanner` fetches `/api/orders` (unfiltered) every 30s and filters client-side. Add a server-side `?status=active` filter or a dedicated lightweight endpoint.
3. **Stale rider location threshold is generous** — 5-minute staleness window (`5 * 60_000`) means the UI can show a rider position that's significantly outdated. Consider reducing to 2 minutes or displaying a "location stale" indicator.
4. **Delivery OTP shown for `READY_FOR_DELIVERY`** — OTP is exposed when the rider may not yet have departed the store. Verify whether this is intentional or should be gated to `OUT_FOR_DELIVERY`/`ARRIVING` only.
5. **HapticButton vibration on unsupported platforms** — `navigator.vibrate(10)` is a no-op on iOS Safari (API not supported). The component works fine without it, but the naming sets an expectation that won't be met on iPhone.

## SSE lifecycle and reconnection gap

The `LiveOrderTracking` component opens an `EventSource` on mount and closes it on the first error with no attempt to reconnect:

```tsx
source.onerror = () => {
  source.close();
};
```

SSE connections drop routinely (mobile network switches, server deploys, idle timeouts). The native `EventSource` API has built-in reconnection behavior when the server sends a `retry:` header, but calling `source.close()` explicitly kills that mechanism. The result is that a user who loses connectivity for a few seconds will never receive another status update until they navigate away and back. For a page whose primary purpose is real-time delivery tracking, this is a meaningful gap. Either remove the explicit close (let the browser's default reconnect handle it) or implement manual reconnection with backoff.

The `LiveOrderBanner` component avoids this problem by polling, but it introduces its own concern: it fetches the full `/api/orders` response (all orders for the user) every 30 seconds and scans client-side for an active status. For a user with dozens of past orders, this is unnecessary payload. A lightweight `/api/orders/active` endpoint or a query-string filter would keep the polling cost constant.

## Rider location staleness and ETA drift

The server-side page guards against stale location data with a 5-minute threshold:

```tsx
Date.now() - order.deliveryPartner.locationUpdatedAt.getTime() < 5 * 60_000
```

Five minutes is generous for a moving rider. If location stops updating (rider app crashes, phone dies), the client will continue showing the last known position and a stale ETA for up to 5 minutes before the server stops sending location data entirely. A "last updated X seconds ago" indicator or a shorter threshold (2 minutes) would signal to the user that tracking data may not be current.

## OTP exposure window

The server page exposes `deliveryOtp` for statuses `OUT_FOR_DELIVERY`, `READY_FOR_DELIVERY`, and `ARRIVING`. `READY_FOR_DELIVERY` means the order is packed and waiting at the store — the rider may not even be assigned yet. Showing the OTP this early creates a longer window during which the code is visible on screen. If the design intent is to show OTP only when the rider is en route, remove `READY_FOR_DELIVERY` from the condition.
