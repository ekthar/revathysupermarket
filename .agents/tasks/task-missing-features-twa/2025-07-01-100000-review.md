# Customer app feature expansion and TWA packaging

A large-scope change adding 14 missing customer-facing features to the Next.js web app — subcategories, reviews, reorder, delivery scheduling, tips, instructions, express checkout, savings summaries, notification granularity, i18n, location-first flow, live tracking enhancements, smart suggestions — then wrapping the web app as a Trusted Web Activity (TWA) Android package with a GitHub Actions CI pipeline, and finally producing a gap analysis between the web staff apps and the React Native mobile-staff app.

The approach is additive: new Prisma models (SubCategory, Review), new API routes with proper auth/validation, new client components wired to existing cart/order infrastructure, next-intl for multilingual support, and a standalone Gradle project for the TWA shell. The TWA is minimal by design (just a Chrome Custom Tab launcher), keeping build complexity low.

**Watch for:** Placeholder fingerprint in Digital Asset Links means the TWA won't pass verification until a real signing key is deployed (confirmed). The express checkout bypasses address/payment selection UI, trusting server-side eligibility — but the eligibility API doesn't validate delivery radius against the stored address coordinates (likely). The `order-checkout.ts` tip handling accepts any numeric value with no upper-bound cap (confirmed). The i18n request handler uses cookie-based locale detection with a `getRequestConfig` that falls back to `en.json` on import failure, meaning a corrupted translation file silently degrades to English without logging (confirmed).

---

## High-level view

The schema layer adds SubCategory (nested under Category) and Review (one per user per product, with a unique composite index). The Order model gains `tipAmount` and `deliveryInstructions` fields. All additive — no destructive changes to existing models.

The API surface grows by seven route files: subcategory CRUD behind admin auth, a public subcategory list, product reviews (GET paginated + POST with purchase verification), product suggestions via co-purchase analysis, cart-level suggestions, a reorder endpoint that checks current stock, and an express-eligibility check.

The checkout flow threads tip and delivery instructions through form state, validation schema, and order creation. No upper-bound validation exists on tip amount beyond `min(0)`.

The TWA is a standard Android Browser Helper project targeting SDK 34, with a GitHub Actions workflow that builds debug APKs unconditionally and release APKs only when keystore secrets are present. The `assetlinks.json` carries a placeholder SHA-256 fingerprint that must be replaced post-signing.

The i18n integration uses `next-intl` with a cookie-based locale persisted client-side. The language switcher fires a PATCH to `/api/user/language` — an endpoint that does not exist in this diff, so the `preferredLanguage` User field is never written server-side.

The gap analysis report is a 570-line document cataloguing 50 feature comparisons across admin, delivery, and packing roles, with a phased implementation plan.

<summary>Issues (8)</summary>

1. **TWA placeholder fingerprint** — `assetlinks.json` contains `__SIGNING_KEY_SHA256__` which will fail Android app link verification. Replace with actual certificate fingerprint after generating the release keystore.
2. **Tip amount unbounded** — `checkoutSchema` validates `tipAmount` as `z.coerce.number().min(0)` with no max. A user could submit an arbitrarily large tip that gets added to the order total. Add a reasonable upper bound (e.g., `.max(500)`).
3. **Express checkout skips radius validation** — The `/api/cart/express-eligible` route checks that a default address exists with coordinates, but never validates whether those coordinates are within the delivery radius. A user who moved could place an undeliverable express order.
4. **Missing `/api/user/language` endpoint** — The `LanguageSwitcher` component fires a PATCH to `/api/user/language` which doesn't exist in this diff. The request silently fails (caught with `.catch(() => {})`), so the `preferredLanguage` field added to the User model is never written.
5. **Translation file import failure silent** — In `i18n/request.ts`, the dynamic `import()` wrapped in try/catch falls back to English without any logging. A typo in a translation file would silently regress the entire locale.
6. **Co-purchase suggestion query performance** — Both `/api/products/[slug]/suggestions` and `/api/cart/suggestions` perform unconstrained `groupBy` on `orderItem` without a date window. On a store with long history, these queries scan all historical order items. Consider adding a `createdAt` filter to the order lookup.
7. **Review upsert race condition** — The purchase verification (`findFirst` on `orderItem`) could theoretically allow a review submission between order delivery and product deletion. Low risk given `onDelete: Cascade` handles cleanup, but the verification and upsert are not wrapped in a transaction.
8. **No test coverage for new API routes** — Seven new API route files and five new client components with no corresponding test additions.

## TWA packaging and Digital Asset Links

The `assetlinks.json` served at `apps/web/public/.well-known/assetlinks.json` contains:

```json
"sha256_cert_fingerprints": ["__SIGNING_KEY_SHA256__"]
```

This placeholder means the TWA will launch in a Custom Tab with a visible URL bar rather than in fullscreen trusted mode until replaced with the actual signing certificate's SHA-256. The workflow conditionally builds a release APK only when `KEYSTORE_BASE64` is set — but there's no step that generates and stores the fingerprint back into the codebase or deployment pipeline. No `gradle-wrapper.jar` is committed (only `gradle-wrapper.properties`), so the workflow relies on the Gradle wrapper download step succeeding at build time.

## Tip and delivery instructions threading

The checkout form threads `tipAmount` and `deliveryInstructions` through the request body to `createAuthoritativeOrder`:

```typescript
const tip = data.tipAmount ?? 0;
const total = Math.max(0, subtotal - discount + deliveryFee + tip);
```

The schema validates `tipAmount: z.coerce.number().min(0).optional()` — no maximum. A malicious client could submit `tipAmount: 99999` and inflate the order total (relevant for COD orders where the store bears the delivery cost expectation mismatch). The client-side `TipSelector` allows arbitrary custom values through a text input capped at 4 characters (max 9999), but server-side validation should be the guard.

## Express checkout eligibility gap

The `/api/cart/express-eligible` route checks: user has at least one delivered order, has a default address with lat/lng, and extracts the last-used payment method. It does **not** check whether those coordinates fall within the store's delivery radius (`SITE.deliveryRadiusKm`). The normal checkout flow validates this client-side via the location prompt, but express checkout skips that entirely. A customer who set their default address within radius, then moved, would create an undeliverable express order.

## i18n dead-end and silent fallback

The `LanguageSwitcher` sets a cookie (`locale=ml`) and calls `router.refresh()` — this works for the next-intl server-side loading. But it also fires `PATCH /api/user/language` which doesn't exist, so the User model's `preferredLanguage` field is never persisted. On next login from a different device, the preference is lost.

The `getRequestConfig` in `i18n/request.ts` wraps the dynamic import in try/catch and falls back to English. No `console.error` or logging. A broken `ml.json` (syntax error, missing key) would silently serve English to Malayalam-speaking users with no signal in monitoring.

## Suggestion queries and scaling

Both suggestion endpoints find orders containing target products, then group co-purchased items by frequency. Neither filters by order date. The `take: 100/200` on the initial `findMany` bounds the number of orders considered, but the subsequent `groupBy` still processes all items in those orders.

For a supermarket processing 200+ orders/day, after six months this means scanning items from orders placed long ago where product catalog may have changed significantly. Adding `where: { order: { createdAt: { gte: thirtyDaysAgo } } }` would improve both relevance and query cost.

## Notification preferences without backend

Three new boolean columns on `UserSettings` (`priceDropAlerts`, `weeklyDeals`, `deliveryAlerts`) default to `true`. The toggles are wired end-to-end through the settings API. However, no notification-sending infrastructure reads these fields — they're preference toggles for a system that doesn't yet exist. This is fine as forward preparation, but users toggling these off today have no effect.
